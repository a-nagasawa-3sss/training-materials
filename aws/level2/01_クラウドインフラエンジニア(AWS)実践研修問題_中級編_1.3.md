## 概要

中級編では、以下の内容を実施します。

- Ruby on Rails のアプリケーションをコンテナ化し、ECS on EC2 および ECS on Fargate を使用したアプリケーションのデプロイとスケーリングを行う
- CodePipeline を利用してビルドとデプロイのプロセスを自動化する
- CloudFront と WAF を導入したキャッシュレイヤーを実装する

最終的には次の構成図で示すようなアプリケーション環境を構築します。

### 最終的なアーキテクチャ（全体像）

```

|  |  |  |  | Public subnet  |  |   |  | Public subnet  |  |      |  ECR     |    | |
```

## タスク 1. コンテナイメージの作成と実行

### タスクの GOAL

- [x] VS Code Server 上でコンテナイメージを実行する
- [x] タスクの完了後は以下のように実行されているアプリケーションを確認できます

（コンテナとして起動した Rails アプリケーションのウェルカム画面が表示され、Rails / Rack / Ruby のバージョン情報を確認できます）

### タスクの流れ

1-1. コンテナイメージを作成する
1-2. コンテナイメージをビルドして実行する

### タスクの要件

#### 1-1. コンテナイメージを作成する

- Docker コマンドを実行してインストールされている Docker のバージョンを確認する
- Docker コマンドを実行して Docker のステータスが「active (runnning)」になっていることを確認する
- コンテナイメージの定義を行う以下の内容の Dockerfile を指定のディレクトリに作成する

`/handson/Dockerfile`

```dockerfile
# Rubyの公式Dockerイメージ (ruby:3.2.1) を取得する
FROM public.ecr.aws/docker/library/ruby:3.2.1

# 必要なパッケージ群を取得する
RUN apt-get update -qq && \
      apt-get install -y nodejs postgresql-client npm && \
      rm -rf /var/lib/apt/lists/*

# ローカルにあるファイルをコンテナイメージ内にコピーする
WORKDIR /myapp
COPY Gemfile /myapp/Gemfile

# Rails アプリケーションを作成する
RUN bundle install && \
      rails new . -O && \
      sed -i -e "52a\ config.hosts.clear\n config.web_console.allowed_ips = '0.0.0.0/0'\n config.action_dispatch.default_headers.delete('X-Frame-Options')" config/environments/development.rb

# Rails を 3000 番ポートで起動する
EXPOSE 3000
CMD ["rails", "server", "-b", "0.0.0.0", "-p", "3000"]
```

- Ruby プロジェクトでライブラリ管理を行うために以下の内容の Gem ファイルを指定のディレクトリに作成する

`/handson/Gemfile`

```ruby
source 'https://rubygems.org'
gem 'rails', '7.2.2'
```

#### 1-2-1. コンテナイメージをビルドして実行する

- Dockerfile をビルドして「rails-app」という名前の Docker イメージを作成する
- 最新の「rails-app」イメージからコンテナを作り、ホストのポート 8081 をコンテナの 3000 ポートにマッピングしてバックグラウンドで実行する
- 画面中央下部の「PORTS」タブを開いて「Port: 8081」に対応する Forwarded Address を別タブで開いて Rails アプリケーションが実行されていることを確認する

## タスク 2. VPC リソースの作成

### タスクの GOAL

- [x] VPC、サブネット、NAT ゲートウェイ、セキュリティグループを構築する
- [x] 後続タスク（ECS サービス作成/ALB 作成）で迷わないよう、必要な SG を用途別に準備する。

### タスクの流れ

2-1. VPC、サブネット、NAT ゲートウェイを作成する
2-2. セキュリティグループを作成する（用途別に 3 つ）

### タスクの要件

#### 2-1. VPC とサブネットを作成する

**VPC**

- リージョンは東京（ap-northeast-1）を選択する
- VPC の名称は「ecs-vpc」とする
- CIDR ブロックは 10.0.0.0/16 とする
- 「ap-northeast-1a」と「ap-northeast-1c」の２つのアベイラビリティゾーンを利用する

**サブネット**

- パブリックサブネットを２つ作成する
  - CIDR ブロックは「10.0.0.0/20」「10.0.16.0/20」とする
  - サブネットに関連付けられたルートテーブルからインターネットゲートウェイへの経路があること
- プライベートサブネットを２つ作成する
  - CIDR ブロックは「10.0.32.0/24」「10.0.48.0/20」とする
  - サブネットに関連付けられたルートテーブルから NAT ゲートウェイへの経路があること

#### 2-2. セキュリティグループを作成する（用途別に 3 つ）

##### 2-2-1. ALB 用セキュリティグループを作成する（ALB 本体はタスク 4、5 で作成）

| 設定項目 | 値 |
|---------|-----|
| SG 名 | `alb-sg` |
| 説明 | Security group for ALB in ecs-vpc |
| VPC | `ecs-vpc` |
| インバウンド | HTTP / TCP 80 / ソース：Anywhere-IPv4（0.0.0.0/0） |
| アウトバウンド | デフォルト（All traffic） |
| 目的 | インターネット → ALB（80）を許可する |

##### 2-2-2. ECS タスク用セキュリティグループを作成する（ECS サービスはタスク 4、5 で作成。サービス作成時にタスク（ENI）へ付与）

| 設定項目 | 値 |
|---------|-----|
| SG 名 | `ecs-task-sg` |
| 説明 | Security group for ECS tasks (allows inbound app port from ALB only) |
| VPC | `ecs-vpc` |
| インバウンド | カスタム TCP / TCP 3000 / ソース：alb-sg（ALB の SG） |
| アウトバウンド | デフォルト（All traffic） |
| 目的 | ALB からタスク（Rails:3000）への通信を許可し、アプリへのルーティングおよびヘルスチェックが正常に行えるようにする。 |
| 補足 | Rails の待受は 3000（Dockerfile/タスク定義のコンテナポートが 3000） |

##### 2-2-3. ECS コンテナインスタンス（EC2）用セキュリティグループを作成する（EC2 コンテナインスタンスはタスク 3-1 で作成。クラスター作成時に EC2 コンテナインスタンスへ付与）

| 設定項目 | 値 |
|---------|-----|
| SG 名 | `ecs-container-instance-sg` |
| 説明 | Security group for ECS container instances (EC2 hosts) |
| VPC | `ecs-vpc` |
| インバウンド | なし（SSH 不要なら閉じる） |
| アウトバウンド | デフォルト（All traffic） |
| 目的 | ECS 用 EC2 ホスト（コンテナインスタンス）の管理・運用のための SG（※ホスト EC2＝タスク/コンテナを載せて実行する土台となる EC2 インスタンス）。本 SG はホスト EC2 側の通信範囲（主にアウトバウンド）を管理するために利用する。 |
| 補足 | タスク用 SG（ecs-task-sg）はタスク（ENI）に付与してアプリの受信（例：3000）を制御するのに対し、本 SG はホスト EC2 側の通信（主にアウトバウンド）を制御する。実運用ではアウトバウンドを必要な宛先のみに絞る設定（例：ECR/CloudWatch Logs/SSM、アプリが接続する DB や外部 API など）とすることが推奨。 |

### タスクの流れ

3-1. ECS クラスターを作成する
3-2. ECR レポジトリを作成してコンテナイメージを保存する
3-3. タスク定義を作成する

### タスクの要件

#### 3-1. ECS クラスターを作成する

- 名前は「ecs-cluster」とする
- クラスターは Fargate と EC2 の両方に対応する
- EC2 をスケールさせる AutoScaling グループは新たに作成する
- プロビジョニングモデルはオンデマンドとする
- AMI は「Amazon Linux 2023」を利用する
- EC2 インスタンスタイプは t3.medium を利用する
- EC2 インスタンスロールは適切なものを設定する
- 必要な容量は最小/最大で１とする
- ネットワーク設定は以下を利用する
  - VPC： ecs-vpc
  - サブネット： ２つのパブリックサブネット
  - セキュリティグループ： ecs-container-instance-sg
  - パブリック IP の自動割り当て： オン

#### 3-2. ECR レポジトリを作成してコンテナイメージを保存する

- ECR を利用してプライベートレポジトリを「rails-app」という名前で作成する
- VS Code Server で作成したイメージを ECR にプッシュする

#### 3-3. タスク定義を作成する

- ECS にワークロードをデプロイするためのタスク定義を作成する
- 名前は「ecs-task」とする
- 起動タイプは Fargate と EC2 の両方に対応する
- 予約する CPU は「.25 vCPU」、メモリは「.5GB」とする
- タスクロールは利用しない
- タスク実行ロールは適切なものを設定する
- コンテナ設定は以下を利用する
  - 名前：rails-app
  - イメージ URI: ECR へプッシュしたイメージ
  - コンテナポート: Rails アプリケーションが待ち受けているポート
  - ログ収集の使用は行わない

### タスクの要件

#### 4-1. サービスを作成する

- サービスの名前は「ecs-ec2-service」とする
- タスク定義は「ecs-task」の最新のものを利用する
- クラスターは「ecs-cluster」を利用する

- 正常性とメトリクスタブ内にある ALB 名をクリックして ALB の DNS 名を確認する
- 確認した ALB の DNS 名に http でアクセスを行いアプリケーションが表示されることを確認する
